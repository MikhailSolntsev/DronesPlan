## Требования к Jira Snapshot MVP

### Функциональные требования

- **Загрузка задач из Jira**
  - Система должна забирать из Jira только вручную добавленные в список задачи. Никакого фильтра, просто задачи по одной
  - Для каждой задачи нужно забирать: ключ, заголовок, статус, assignee, тип задачи, ссылку на задачу.

- **Расчёт приоритета**
  - Для каждой задачи система вычисляет приоритет 0–100 на основе статуса и, при необходимости, типа задачи.
  - Маппинг статусов и типов в приоритеты задаётся в конфигурации и может изменяться без перекомпиляции.

- **Фильтрация закрытых задач**
  - Задачи со статусом, соответствующим приоритету 100 (закрыто), не отображаются в активном списке.

- **Снапшоты**
  - При каждой синхронизации создаётся батч снапшота (SnapshotBatch) с отметкой времени.
  - Для каждой задачи в батче сохраняется IssueSnapshot.

- **Отображение списка задач**
  - UI должен показывать список всех задач из текущего (последнего) батча.
  - Список сортируется по приоритету по убыванию, затем по ключу задачи.
  - В таблице отображаются минимум: приоритет, ключ, заголовок, статус, assignee, время снапшота, ссылка на Jira.

- **Календарный/ленточный вид**
  - UI должен показывать упрощённый календарь: по строкам задачи или сотрудники, по столбцам дни выбранного периода.
  - В ячейке для текущего дня отображается текущий assignee по данным последнего снапшота.
  - Цвет ячейки определяется статусом/приоритетом (разработка/тестирование/прочее).
  - Календарь должен быть на 2 недели вперёд
  - Выходные дни (сб-вс) отображаются одной колонкой, выделенной морковным цветом 

- **История по неделям**
  - Система должна позволять выбрать неделю (или дату), чтобы посмотреть состояние задач на тот момент (по сохранённым батчам).

- **Ручное редактирование**
  - Пользователь может вручную задать исполнителя на определённый день как планируемого 
  - Приоритет может редактироваться вручную в таблице (в рамках 0-49), чтобы приоритезировать бэклог

### Нефункциональные требования

- **Простота и предсказуемость**
  - Система не должна пытаться прогнозировать даты завершения задач или автоматически планировать загрузку сотрудников.
  - Роль системы — дать актуальный и исторический срез фактического состояния задач и исполнителей, а также заданный пользователем план

- **Расширяемость**
  - Архитектура должна позволять добавить в будущем:
    - учёт отпусков;
    - отчёты по отдельным сотрудникам;
    - дополнительные визуализации.

- **Инфраструктура**
  - Приложение развёртывается в докере на локальной машине
  - Доступ к Jira осуществляется по REST API on‑premise инстанса.
