## Как агенту работать в этом репозитории

- **Технологии**: основной стек — C# и .NET (ASP.NET Core). Предпочтительно следовать типичной многослойной архитектуре (Domain / Infrastructure / Web).
- **Источник данных**: Jira развёрнута on‑premise, доступ по прямому REST API. Любые данные о задачах считаются вторичными по отношению к Jira.
- **Назначение проекта**: не \"умная\" система планирования, а тупое зеркалирование текущих статусов и assignee из Jira в удобный для тимлида вид (приоритеты 0–100, календарь).

### Что важно учитывать

- **Приоритеты 0–100**:
  - 0–49: задача ещё не началась (\"бэклог\" в разных состояниях: Открыто, Анализ, Готово к разработке и т.п.).
  - 51: В работе (разработка).
  - 61: Ревью.
  - 69: Готово к тестированию.
  - 71: Тестирование.
  - 81: Сборка.
  - 91: Релизное тестирование.
  - 99: Релиз.
  - 100: Закрыто (не должно отображаться в активном списке).

- **Не пытаться угадывать планирование**:
  - Система не должна сама придумывать даты завершения задач.
  - Все прогнозы и плановые диапазоны дат остаются в зоне ответственности тимлида.

- **Расширяемость**:
  - Проектируется сразу с учётом возможного добавления: отдельного просмотра по сотрудникам, учёта отпусков, дополнительных метрик.
  - Слои Domain/Infrastructure/Web должны быть чётко разделены, чтобы можно было менять реализацию хранения или Jira-клиента без ломки домена.

### Где хранится контекст

- Общие описания и цели: `.cursor/PROJECT.md`.
- Настоящий файл: высокоуровневые инструкции для агента.
- Правила и стандарты кода и домена: `.cursor/rules/*.mdc`.
- Продуктовые требования и маппинги: `.cursor/specs/*.md`.
- Техническая архитектура и детали интеграции: `.cursor/tech/*.md`.
- Чек‑листы задач для реализации: `.cursor/tasks/todo.md`.
